§1 SOLID Principles — Where Each Lives in This Code
S
Single Responsibility
Each class does ONE thing. Config holds config. Processor processes. Validator validates. No god classes.
PaymentConfig → only config
CreditCardProcessor → only CC
EmailNotifier → only email
💬 "PaymentConfig doesn't validate or charge — it only holds gateway URLs and timeouts."
O
Open / Closed
Add new payment types WITHOUT modifying existing processors. Extend behavior, don't edit working code.
// Add CRYPTO:
class CryptoProcessor
  implements PaymentProcessor
// + 1 factory case. Done.
💬 "When AT&T added crypto, we added one class + one factory case. Zero changes to CC or ACH code."
L
Liskov Substitution
Any PaymentProcessor subtype is safely substitutable. ACH in place of CC? The orchestrator doesn't break.
PaymentProcessor p = factory
  .create("ACH");
p.charge(req); // works fine
💬 "ACH, CC, Crypto — any is substitutable behind PaymentProcessor. The orchestrator never cares which."
I
Interface Segregation
No fat interfaces. Chargeable, Refundable, Tokenizable are separate. ACH doesn't stub tokenize().
CC → Chargeable + Refundable
     + Tokenizable
ACH → Chargeable + Refundable
     (no Tokenizable!)
💬 "ACH doesn't implement Tokenizable — it can't tokenize a bank account the way you tokenize a card."
D
Dependency Inversion
High-level modules depend on abstractions. Orchestrator → PaymentProcessor interface, never a concrete class.
// Orchestrator depends on:
PaymentProcessor (interface)
// NOT on:
CreditCardProcessor (concrete)
💬 "We swapped CyberSource→Fiserv without touching the orchestrator — it depends on abstractions."

CREATIONAL How objects are created
PATTERN 01
Singleton
PaymentConfig
What: One instance, globally accessible, thread-safe (Bill Pugh holder idiom).
Why: Multiple config objects = contradictory gateway URLs. One source of truth.
💬 "Singleton for gateway config — one source of truth for Fiserv URLs, merchant IDs, timeouts."
PATTERN 02
Builder
PaymentRequest.Builder
What: Fluent, step-by-step construction of a complex immutable object.
Why: 8+ fields. A constructor with 8 params is unreadable. Builder validates before build().
💬 "PaymentRequest has 8 fields — amount, token, merchant ID. Builder makes it readable and validates before the object exists."
PATTERN 03
Factory Method
PaymentProcessorFactory
What: Static method decides which concrete class to instantiate by type.
Why: Caller says "CREDIT_CARD" → gets CreditCardProcessor. Never uses new directly.
💬 "Adding crypto? One new class + one factory case. Existing CC and ACH code untouched — Open/Closed."
PATTERN 04
Abstract Factory
GatewayFactory
What: Creates a family of related objects (processor + tokenizer + settlement).
Why: Fiserv processor MUST pair with Fiserv tokenizer. Mixing families = runtime disaster.
💬 "When we switched CyberSource→Fiserv, we swapped ONE factory. Every downstream got the new family — no mix-ups."
STRUCTURAL How objects are composed
PATTERN 05
Adapter
LegacyCyberSourceAdapter
What: Makes an incompatible old interface work with a new one.
Why: CyberSource SOAP client has runTransaction(soapEnvelope). New system expects charge(PaymentRequest). Adapter translates.
💬 "During migration, Adapter let the new REST orchestrator talk to legacy SOAP without changing either side."
PATTERN 06
Facade
PaymentFacade
What: Simplified entry point to a complex multi-step subsystem.
Why: WEBCAPM "Save" button → one endpoint → facade orchestrates validate→fraud→tokenize→charge→settle.
💬 "React frontend calls ONE endpoint. Facade orchestrates 6 steps behind the scenes."
PATTERN 07
Decorator
LoggingDecorator
RetryDecorator
What: Wraps an object to add behavior without modifying its code. Composable.
Why: retry(logging(processor)) — each adds one concern. Processor code stays clean.
💬 "Logging and retry are decorators, not if-blocks inside the processor. They compose: retry wraps logging wraps processor."
PATTERN 08
Proxy
TokenizationCacheProxy
What: Controls access to a real object — adds caching, lazy-loading, or access control.
Why: Same card tokenized repeatedly in AutoPay re-enrollment. Proxy caches → no redundant gateway calls.
💬 "Proxy caches tokenization so repeated calls for the same card don't hit Fiserv again — critical for AutoPay."
BEHAVIORAL How objects communicate
PATTERN 09
Strategy
FraudStrategy
What: Family of interchangeable algorithms. Swap at runtime without changing the caller.
Why: High-value → ML-based fraud check. Low-value → velocity check. Engine doesn't hardcode which.
💬 "Fraud strategies are pluggable — velocity for small, ML for large. Engine receives its strategy, doesn't choose it."
PATTERN 10
Observer
PaymentEventPublisher
What: When something happens, notify all interested listeners automatically.
Why: Payment approved → email + ledger + dashboard + settlement, all independently subscribed. Same idea as Kafka pub/sub.
💬 "Observer decouples events from actions. At infra scale, this is exactly what our Kafka pipelines do."
PATTERN 11
Template Method
AbstractSettlementProcessor
What: Defines algorithm skeleton in base class. Subclasses override specific steps, not the flow.
Why: Settlement always: validate→prepare→execute→confirm. Fiserv/CyberSource override steps, not sequence.
💬 "Template Method locks the settlement sequence. Fiserv and CyberSource override steps, never the flow."
PATTERN 12
Chain of Responsibility
ValidationHandler chain
What: Request passes through a chain of handlers. Each passes or rejects.
Why: Amount→Token→Merchant→Idempotency. Each independent, composable, reorderable.
💬 "Validation is a chain — amount, card token, merchant whitelist, idempotency. Reorder or extend without touching handlers."
